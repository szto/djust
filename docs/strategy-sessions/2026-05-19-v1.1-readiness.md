## Recommendation (Stage 6)

**Path E — Defer to launch soak.**

Reasoning (load-bearing):

1. **The user invocation quoted the contingency, not the recommendation.** The brainstorm hierarchy was Path A > Path B, but it explicitly carried *"Path B is the safer recommendation if 1.0 launch reveals adoption friction."* The user invocation quoted that exact contingency, signaling the load-bearing question isn't *"A or B?"* — it's *"is the contingency triggered?"* — and that question can't be answered without launch data.

2. **The cost of Path E is small.** ~2-3 weeks of pre-reqs + cleanup that any headline path needs anyway (H5 docs versioning, H1 wire-protocol coverage, D10 inline benchmarks, D8 audit-as-GHA, E1 DRF interop). Nothing is wasted; everything composes with whichever headline path the post-soak re-strategize chooses.

3. **Committing now without data is the worse choice.** Picking Path A or Path B before 1.0 launch feedback locks in an answer to *"what does the 2026 Python web community want from djust?"* without consulting the only data source that can answer it. The active ADRs (002 ph4-5, 003-006) explicitly mark the AI arc as post-1.0 — they support deferring the commitment, not rushing it.

4. **Optionality has real value.** If post-soak data strongly favors one of A/B/C/D, the post-soak re-strategize is a 1-stage decision rather than a 6-stage commitment. If post-soak data is ambiguous, the brainstorm's default (Path A) becomes the headline — but that decision is made *with* the option to override based on new information, which is strictly better.

**ADR alignment:** ADRs 002 ph4-5, 003, 004, 005, 006 mandate execution post-1.0, not execution in 1.1 specifically. Path E respects that latitude. No active ADR is overridden. No directional change. No new ADR required.

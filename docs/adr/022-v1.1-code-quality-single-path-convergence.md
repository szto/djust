## Context

The v1.0.x arc (v1.0.0 GA → v1.0.8) shipped 8 patch milestones plus the v1.0.7 security audit + v1.0.8 prevention program. Across that arc, **one failure class dominated every retro: `#1646` parallel-path-drift** — a control or behavior implemented per-transport (WebSocket / SSE / `ViewRuntime` / HTTP-API / HTTP-GET) that drifts when one path is changed and a parallel path is not. It appears **21× in `RETRO.md` and 12× in `CLAUDE.md`**, recurred **3× within the single v1.0.8-2 drain** and **4×** the cycle before, and was the root shape of the entire security audit (the systemic finding "SSE consistently lacks controls the WS transport has").

The v1.0.8-1 prevention program converged the *auth sequence* into one `run_pre_mount_auth` helper and shipped anti-drift nets (`#1859` transport-parity suite + mount-orchestration AST pins), but **deliberately deferred the full dispatch convergence** — migrating WS `handle_mount` + `_handle_event_inner` (~1000 lines, ~16 WS-only behaviors) and the SSE mount/event paths onto a single `ViewRuntime.dispatch` spine — soak-gating it under ADR/`#1122` (split-foundation) because it is the single highest-blast-radius change in the codebase. That deferral (private `#173` WU3/WU4) is now the residual root of the `#1646` class: as long as N dispatch paths exist, drift is *possible*; only collapsing them to one makes it *impossible*.

The user, at the v1.1 milestone boundary, steered v1.1 toward **code quality / DRY / single-path / refactor**. This ADR records the resulting directional decision.

## Decision

**v1.1's headline is a code-quality / single-path convergence arc, executed quality-first (strategy Path B):**

1. **Quality groundwork first (low-risk, continuous, builds the safety net).** Extend the anti-drift parity nets to every control; pay down concrete DRY/footgun debt (`#1360` dedupe `_parseTimeMs`/`_computeTransitionTiming`, `#1279` single `isSignificantChild`, `#1368` log-injection asymmetry in `rust_bridge`, `#1648` sync-async `check_handler_permission`, `#1356` state-backend shared-ref); canonicalize the opt-in framework patterns (`#1307`) + finish the bidirectional-binding inventory (`#1308`); de-flake the suite + promote the browser-smoke to a hard gate (`#1875`, `#1869`, `#1152`, `#1153`).
2. **Then the ViewRuntime convergence headline.** Once drift is *mechanically detectable* (the nets from step 1), migrate the WS + SSE dispatch onto the single `ViewRuntime` spine in split-foundation iterations (mount first, then event), retiring the `#1646` class structurally. Module-organization refactors (splitting `websocket.py` ~5000 lines / `live_tags.py`) ride alongside this step, not before it (reorganizing before converging would move the same code twice).

**Why quality-first and not convergence-first:** the convergence is the right destination, but landing a ~1000-line dispatch merge with ~16 WS-only behaviors *without* the parity nets in place is the exact "giant un-netted PR" risk that `#1122` (split-foundation) and `#1859` (anti-drift nets) exist to prevent. The v1.0.8-1 convergence was safe *because* the nets came first; v1.1 repeats that discipline at larger scope.

## Consequences

- **Positive**: retires the #1 recurring failure class of the v1.0.x arc at its root; each early iteration delivers a standalone quality win (no "big bang" dependency); the convergence lands on a net, not a hope; one dispatch spine means a future control is added once and lives on every transport.
- **Cost / risk**: the convergence step remains the highest-blast-radius change in the codebase — mitigated by (a) doing it last, after the nets, and (b) split-foundation iterations with per-iteration soak per `#1122`. The ~16 WS-only behaviors (binary upload, presence, cursor, time-travel, sticky-child preservation, signed-snapshot restore, actor channel-layer) must each be explicitly mapped onto runtime hooks — this is the work the v1.0.8-1 retro flagged as "not a thin shim."
- **Relationship to parallel ADRs**: ADR-019 (LiveView Native) Iter-I is *renderer abstraction + `ViewRuntime` plumbing* — it needs the same single `ViewRuntime` spine this convergence builds, so the two are **aligned, not competing**; sequencing the convergence first de-risks LVN-I. ADR-020 (`dj-island`) and ADR-021 (auto-navigation) are independent feature tracks and are unaffected.
- **Revisit trigger**: if an early-iteration parity-net pass reveals the dispatch paths have already diverged in ways that make convergence cheaper to do *first* (i.e., the reorg/DRY work is fighting the fork), re-run `/pipeline-strategy` to reconsider Path A vs B ordering.

## Convergence sequencing — refined by Stage-4 scoping (2026-06-23)

A read-only Stage-4 investigation (the "first-principles grep before architecting" canon) scoped the convergence before any code and **partially tripped the revisit trigger** — direction is sound (proceed iteration-by-iteration; do NOT re-strategize) but the original "WS-mount first" sequencing was wrong. Findings:

- **`ViewRuntime.dispatch_mount` (`runtime.py:303`) / `dispatch_event` (`runtime.py:512`) already exist but have ZERO production call sites** — only `dispatch_message` is wired (WS `url_change` `websocket.py:4824`; SSE `/message/` `sse.py:1192`). They are an unexercised **fourth** parallel copy that has **already drifted** from the WS handlers.
- **The runtime path is missing controls the WS path has** — `dispatch_mount` lacks the post-mount `check_object_permission` (a latent IDOR-class gap, not currently live because nothing mounts through it), and the runtime drains **3 of WS's 8 flush queues** (`runtime.py:1038` vs `websocket.py:888` `_flush_all_pending`) — a live #1646 instance *inside* the supposed cure.
- **SSE doesn't use the runtime to mount** — the SSE stream GET still mounts via legacy `_sse_mount_view` (`sse.py:996`) and `/event/` via `_sse_handle_event` (`sse.py:1120`). So the runtime is unproven against any real mount/event transport.
- **A thin-shim of WS-mount is INFEASIBLE** (confirmed #1853): the mount delta is ~20 behaviors, ~8 of them transport-agnostic state the runtime lacks. Convergence must be `run_pre_mount_auth`-style **sequence extraction**, never a thin shim.
- **The parity nets pin security re-forks, not behavioral parity** — they must grow flush-queue-count / object-perm / version-stamping nets BEFORE the migration, or convergence silently inherits the runtime's gaps.

**Refined split-foundation order (supersedes the "WS-mount first" implication above):**

- **Iter 0 — behavioral-parity nets + runtime gap-fixes** (Low risk / S). Add flush-queue-count, object-permission, and wire-version parity nets; fix the runtime's gaps (add the 5 missing flush queues; add `check_object_permission` to `dispatch_mount`); add a count-test enumerating the WS-only behaviors. Each a standalone quality win; closes the latent object-perm gap before anything routes through the runtime.
- **Iter 1 — converge SSE legacy mount/event onto the runtime** (Low-Med / M). SSE has ~no WS-only behaviors, so this proves the runtime against a real transport and forces it to grow the transport-agnostic hooks WS needs later — a soak release before the high-blast WS work. No WS changes.
- **Iter 2 — WS event convergence** (High / L) — **SPLIT into 4 phases** (Stage-4 scoping 2026-06-23). `event` is a single verb (routing gated by `RUNTIME_OWNED_VERBS` membership at `websocket.py:1933`), so it CANNOT be partially flipped — the only safe shape is to grow `dispatch_event` + `WSConsumerTransport` hooks to full parity over zero-WS-routing-risk PRs (each verified against SSE/url_change), then ONE final flip. `_handle_event_inner` is ~1168 lines (`websocket.py:3099`); `_dispatch_event_inner` (`runtime.py:649`) is the clean ~110-line minimal spine (SSE's only event path post-Iter-1), so every WS-only behavior is a genuine ADD (the `run_pre_mount_auth` extraction shape):
  - **Phase 2.0** — runtime event-spine parity nets + cheap shared grows (`ref` echo, `source="event"`, `event_name`-on-noop, `_force_full_html`, `_notify_waiters` ADR-002, #700 identity-push-skip) + an event-behavior enumeration net. Blast radius: SSE only. Low/S. **← first PR.**
  - **Phase 2.1** — move `component_id` (LiveComponent), `view_id` (sticky-child), and embedded-child routing into the runtime (transport-agnostic; the hardened embedded-error path moves verbatim). Med/M. No WS routing change.
  - **Phase 2.2** — move time-travel record, session state-save (#1466), and sticky-child state-save (ADR-018) into the runtime; carry the #1466 source-grep pin to the new location in the SAME PR (#1391 symbol-migration canon); `_maybe_push_tt_event` becomes a `transport.on_event_recorded` hook. Med-High/M-L. No WS routing change.
  - **Phase 2.3 — FINAL FLIP — itself SPLIT into 2.3a (build-up, zero-WS-routing-risk; 2-3 PRs) → 2.3b (atomic flip)** (Stage-4 flip-scoping 2026-06-24). The remaining fold-work is large + has two load-bearing findings, so fold+flip in one PR is the giant-un-netted-PR #1122/#1859 exists to prevent:
    - **TWO load-bearing findings.** (1) **Actor event hook**: `dispatch_event` has NO actor branch (the `use_actors` guard is in `dispatch_mount` runtime.py:483 and refuses); a WS actor view mounts (bespoke, sets `actor_handle`) then post-flip its events hit `dispatch_event` with no actor branch → silently runs the handler in-process, desyncing the actor diff baseline. Fix: `transport.uses_actors(view)` + `transport.dispatch_actor_event(...)` hook (WS runs the block verbatim vs `consumer.actor_handle`; SSE never). (2) **The runtime's `_render_lock` (runtime.py:377) is DEAD CODE — never acquired anywhere.** The real lock is consumer-owned (websocket.py:619) + `_processing_user_event` and is SHARED with the WS-only `_run_tick`/`server_push`/`db_notify` loops. A runtime-local lock is a different object → cannot serialize against ticks → the exact #560 version-interleave bug. Fix: **DELETE runtime.py:377**; borrow the consumer's lock via a `transport.event_context()` async-CM (WS: acquire lock + set `_processing_user_event` + origin-channel + PerformanceTracker/SQL-capture, yield, reset in finally; SSE: no-op).
    - **2.3a** (Med-High): add the actor-event hook; add `transport.event_context()` (render-lock + #1677 origin-channel + observability); GROW `dj_activity` gating + a lock-free deferred-event re-dispatcher in the runtime (the flush's `_dispatch_single_event` assumes the held lock — websocket.py:1467); fix the binary/msgpack framing gap (`WSConsumerTransport.send` must honor `consumer.use_binary` for patch frames — runtime emits JSON-only today); add a `transport.recheck_event_auth` hook for `reauth_on_event` (#1777/T3, mind #291); reconcile the async-work `source="async"` drift (#1646 inside the cure). `RUNTIME_OWNED_VERBS` UNCHANGED, `_handle_event_inner` UNTOUCHED.
    - **2.3b** (High/SMALL diff): add `event` to `RUNTIME_OWNED_VERBS` (websocket.py:4830); reduce `handle_event` to a thin shim like `handle_url_change` (websocket.py:4846); delete `_handle_event_inner`; flip the two boundary pins (test_ws_receive_runtime_dispatch_1852.py:512). Gate-off-verified.
    - **FIRST PR = the 6 real-`WebsocketCommunicator` gap-tests against the CURRENT bespoke path** (#1466/#1780/#1468): `component_id` routing, time-travel-on-event, `dj_activity` deferral, actor-path events, `ref` echo, `_force_full_html`. They characterize what the flip must preserve, pass NOW, and must stay green through the flip = the parity proof. (The 2.1 component/2.2 tt tests exist but use a MockTransport, not a real communicator.) Each asserts intermediate state + has a gate-off sibling. Blast radius of the flip: ALL WS events.
- **Iter 3 — WS mount convergence** (High / L) — **SPLIT 3.0→3.1→3.2→3.3a→3.3b** (Stage-4 mount-scoping 2026-06-24). `mount` is a single all-or-nothing verb (gated by `RUNTIME_OWNED_VERBS` membership at `websocket.py:1985`), so same shape as the event flip: grow `dispatch_mount` (now `runtime.py:1105`) + mount-side `WSConsumerTransport` hooks to a functional superset of `handle_mount` over zero-WS-routing-risk PRs, then ONE atomic flip. The mount delta is LARGER than event (~20 behaviors, ~8 transport-agnostic) with **THREE load-bearing flip findings**:
  - **(A) Stale `runtime.view_instance` idempotency collision** (the #560-class landmine): `dispatch_mount` early-returns if `view_instance is not None` (`runtime.py:1125`); the consumer nulls `self.view_instance` on disconnect/auth-fail/**live_redirect teardown** (`websocket.py:3844`) but NEVER `runtime.view_instance`. Post-flip a `live_redirect_mount` re-mount silently no-ops. Fix: the mount shim + disconnect/teardown reset `runtime.view_instance=None` before dispatching mount.
  - **(B) `view_instance` ownership INVERTS at mount** — url_change/event mirror consumer→runtime; mount CREATES the view (runtime→consumer), so the shim reads back `self.view_instance = runtime.view_instance` and every WS-only post-mount consumer attr (`_view_group`, `_tick_task`, `use_actors`, `actor_handle`, presence/db_notify groups, `_sticky_preserved`) moves into a transport hook that writes onto the consumer during the runtime mount.
  - **(C) Mount wire-version**: WS uses the consumer-owned monotonic `_next_version()` (#1788) and crucially NOT `_next_version_armed` (mount establishes baseline, does NOT arm request_html recovery) — the existing `transport.next_client_version` arms, so mount needs a distinct `transport.next_mount_version()` (WS no-arm; SSE raw).
  - Plus **(D)** actor mount is actively *refused* by the runtime (`runtime.py:1170`) → `transport.uses_actors_for_mount`/`dispatch_actor_mount` hook (WS verbatim; SSE keeps refusing); and **(E)** auth verdict→frame drift (error key `error` vs `message`, missing `close(4403)`, the `_mounting_in_batch`/#291 gate, missing `run_on_mount_hooks`) → `transport.finalize_mount_auth` hook.
  - `handle_mount_batch`/`_mount_one` stay WS-only: the `send_json`→`_collect` swap still captures the runtime's frames (runtime emits via `WSConsumerTransport.send`→`consumer.send_json`), so the collector contract is unchanged; `finalize_mount_auth` gates `close(4403)` on `not _mounting_in_batch` (#291/#1780 — a batched login-required view reports as `navigate[]`, not a socket close that kills siblings; `test_ws_auth_close_socket.py` is the net).
  - **Phase 3.0 (first PR)**: mount-spine parity nets + 6 real-`WebsocketCommunicator` gap-tests vs the bespoke path (actor mount, `sticky_hold` ordering, group_add reachability, tick-at-mount, optimistic/upload on the frame, **live_redirect re-mount idempotency** = the Finding-A net) + cheap transport-agnostic grows (`_djust_mount_request` stash #1895, dirty-baseline, has_prerendered, optimistic/upload, mount-drain with the #1391 pin move). Then 3.1 (state restore + on_mount hooks) → 3.2 (dormant hook scaffolding) → 3.3a (wire hooks into dispatch_mount) → 3.3b (the atomic flip: `mount` into `RUNTIME_OWNED_VERBS` + thin-shim, A/B/C wired). Module-org refactor of `websocket.py` rides AFTER 3.3b (ADR Decision-2). Real-`WebsocketCommunicator` tests mandatory; per-iteration soak (#1122).

Per-iteration soak (#1122) between each. Iter 0 is the executable first PR for v1.1.0-3.
